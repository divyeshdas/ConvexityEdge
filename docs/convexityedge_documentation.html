Chapter 02

Mathematical Foundation

All pricing in ConvexityEdge is based purely on the Black-Scholes-Merton model for European options. No stochastic volatility (Heston), no stochastic interest rates (Vasicek/CIR). This section explains every formula used, from first principles.

2.1 Black-Scholes Model

Inputs

SymbolNameTypical ValueNotes
SSpot priceMarket priceCurrent underlying price
KStrike priceOption contractThe agreed exercise price
TTime to expiryDTE / 365Expressed in years
rRisk-free rate~0.053 (5.3%)US 10-year treasury yield
qDividend yield~0.013 for SPYContinuous dividend yield
σVolatility0.15–0.80Annualized implied volatility

Intermediate Values (d1 and d2)

d₁ = [ ln(S/K) + (r − q + σ²/2)·T ] / ( σ·√T )
d₂ = d₁ − σ·√T

ln(S/K) is the log-moneyness — positive for ITM calls, negative for OTM calls. σ·√T is the expected one-standard-deviation move over the option's life. N(d₂) is the risk-neutral probability that the option expires in-the-money.

Option Prices

Call = S·e−qT·N(d₁) − K·e−rT·N(d₂)
Put = K·e−rT·N(−d₂) − S·e−qT·N(−d₁)

N(x) is the standard normal cumulative distribution function — the probability that a draw from N(0,1) is ≤ x. Implemented as scipy.stats.norm.cdf(x). The exponential discount factors e^(-rT) and e^(-qT) account for the time value of money and foregone dividends.

Put-Call Parity Check
C − P = S·e−qT − K·e−rT
This relationship must hold for any arbitrage-free market. The backend validates this for all computed prices.

Vectorized Batch Pricing

For pricing an entire options chain (300+ contracts), the engine uses NumPy to compute all prices simultaneously rather than looping:

def bs_price_batch(S, K_arr, T_arr, r, q, sigma_arr, types):
    # All arrays — NumPy vectorizes across all 300+ contracts at once
    d1 = (np.log(S/K_arr) + (r - q + 0.5*sigma_arr**2)*T_arr) / (sigma_arr*np.sqrt(T_arr))
    d2 = d1 - sigma_arr * np.sqrt(T_arr)
    call_px = S*np.exp(-q*T_arr)*norm.cdf(d1) - K_arr*np.exp(-r*T_arr)*norm.cdf(d2)
    put_px  = K_arr*np.exp(-r*T_arr)*norm.cdf(-d2) - S*np.exp(-q*T_arr)*norm.cdf(-d1)
    return np.where(types == 'C', call_px, put_px)

Performance: 10,000 contracts priced in ~4ms — well within the 2,000ms target.

2.2 The Greeks

Greeks measure the sensitivity of option price to changes in each input. All are computed analytically (exact closed-form derivatives), not numerically (bumping inputs). Analytical Greeks are faster, exact, and have no finite-difference error.

Delta (Δ) — Price Sensitivity

Call Δ = e−qT·N(d₁) ∈ [0, 1]
Put Δ = −e−qT·N(−d₁) ∈ [−1, 0]

Delta tells you: if the stock moves $1, how much does the option price move? An ATM call has Δ ≈ 0.50. A deep ITM call has Δ ≈ 1.0 (moves dollar-for-dollar). Delta is also the hedge ratio — short 0.5 shares to delta-hedge one long ATM call.

Gamma (Γ) — Delta Sensitivity

Γ = e−qT·φ(d₁) / ( S·σ·√T ) [same for calls and puts]

φ(x) is the standard normal PDF. Gamma peaks at-the-money and near expiry — this is where delta changes fastest and delta-hedging requires the most frequent rebalancing. High gamma = dangerous near expiry.

Vega (ν) — Volatility Sensitivity

ν = S·e−qT·φ(d₁)·√T / 100 [per 1% change in IV, same for calls and puts]

Vega is largest for ATM options and long-dated options. The ÷100 makes it readable: "option price changes by $0.12 per 1% move in IV." Selling premium = short vega (you lose when IV spikes).

Theta (Θ) — Time Decay

Θ_call = [ −S·e−qT·φ(d₁)·σ/(2√T) − r·K·e−rT·N(d₂) + q·S·e−qT·N(d₁) ] / 365

Theta is always negative for long options — you lose time value every day. It accelerates as expiry approaches. Theta and Gamma are a trade-off: high gamma (good for long gamma scalpers) comes with high negative theta (you pay for it daily).

Rho (ρ) — Interest Rate Sensitivity

ρ_call = K·T·e−rT·N(d₂) / 100
ρ_put = −K·T·e−rT·N(−d₂) / 100

Rho is the least important Greek for short-dated options. For long-dated LEAPS, a 1% rate change can meaningfully affect call prices.

2.3 Implied Volatility Solver

Implied volatility (IV) is the reverse problem: given an observed market price, what σ would Black-Scholes need to produce that price? There is no algebraic inverse — we must search numerically.

Initial Guess (Brenner-Subrahmanyam Approximation)

σ₀ ≈ √(2π/T) · (C / S)

This approximation is accurate for ATM options and places the starting point close to the solution, reducing the number of iterations needed.

Newton-Raphson Method (Primary)

σ_{n+1} = σ_n − ( BS(σ_n) − Market Price ) / Vega(σ_n)

Newton's method converges quadratically near the root — typically 3–5 iterations. It fails when Vega is near zero (deep ITM or OTM options where the price is nearly all intrinsic value).

def solve_iv_newton(market_price, S, K, T, r, q, option_type, max_iter=50):
    sigma = sqrt(2*pi/T) * (market_price/S)   # Brenner-Subrahmanyam guess
    for _ in range(max_iter):
        price = bs_price(S, K, T, r, q, sigma, option_type)
        vega  = bs_vega(S, K, T, r, q, sigma)
        diff  = price - market_price
        if abs(diff) < 1e-8: return sigma   # converged
        if abs(vega) < 1e-10: raise ValueError("vega too small")
        sigma -= diff / vega
        sigma = max(0.001, min(sigma, 10.0))   # clamp to [0.1%, 1000%]
    raise ValueError("did not converge")

Brent's Method (Fallback)

Brent's method is a hybrid of bisection, secant, and inverse quadratic interpolation. It guarantees convergence as long as a bracket exists (i.e., the function changes sign over [σ_lo, σ_hi]). We bracket [0.001, 10.0]. It is slower than Newton but never fails.

IV Solver Strategy
Primary: Newton-Raphson (fast, 3–5 iters) → Fallback: Brent's method (slow, guaranteed). The solver chain automatically falls back on any exception (zero vega, non-convergence, numerical overflow).

2.4 IV Surface Construction

IV Smile

Plot IV vs. strike for a single expiry. Calls and puts are plotted separately. In real markets, the smile is not flat — deep OTM puts are more expensive than calls (negative skew) because of crash fear.

IV Term Structure

Plot ATM IV vs. DTE across all available expirations. Normal markets show upward-sloping term structure (more uncertainty further out). Inverted term structure (front month IV > back month) signals a near-term event risk (earnings, FOMC).

IV Rank

IV Rank = ( Current IV − 52-week Low ) / ( 52-week High − 52-week Low ) × 100

A rank of 80 means IV is in the 80th percentile of its past year. High IV Rank is favourable for selling premium (straddles, iron condors). Low IV Rank favours buying options.

3D IV Surface

A 2D grid of (strike, DTE) with IV as the Z-axis. Built by running the IV solver for every contract in the chain, then rendering with echarts-gl's WebGL surface series. The colour map goes from deep blue (low IV) through yellow to red (high IV).

3.2 FastAPI Application

Lifespan Context Manager

FastAPI's lifespan replaces the deprecated @app.on_event startup/shutdown hooks. It uses an async with context manager — code before yield runs at startup, after yield at shutdown.

@asynccontextmanager
async def lifespan(app: FastAPI):
    logger.info("ConvexityEdge backend starting...")
    asyncio.create_task(run_refresh_loop())   # start 60s background refresh
    try:
        await get_redis()                    # warm up Redis connection
    except Exception:
        logger.warning("Redis unavailable — running without cache")
    yield                                    # app is now serving requests
    # shutdown cleanup happens here

ORJSON Responses

All endpoints use ORJSONResponse instead of the default JSON response. ORJSON is 2–10× faster than Python's stdlib json module and natively handles NumPy arrays, datetime objects, and Decimal types.

3.4 Data Pipeline

Market Data Provider (Abstract Pattern)

The MarketDataProvider abstract base class defines the contract for all data sources. YFinanceProvider implements it using yfinance. Switching to a live data source (Polygon.io, Alpaca, IBKR) only requires writing a new provider class.

class MarketDataProvider(ABC):
    @abstractmethod
    async def get_quote(self, symbol: str) -> QuoteData: ...
    @abstractmethod
    async def get_option_chain(self, symbol: str, expiry: date) -> list[OptionContractData]: ...

yfinance Thread Pool Pattern

yfinance makes synchronous HTTP requests. Running these directly in FastAPI's async event loop would block it, preventing other requests from being served. The solution is to run yfinance in a thread pool:

async def get_option_chain(self, symbol, expiry):
    def _fetch():          # synchronous — runs in thread
        ticker = yf.Ticker(symbol)
        chain  = ticker.option_chain(expiry.isoformat())
        # parse calls and puts ...
    loop = asyncio.get_event_loop()
    return await loop.run_in_executor(None, _fetch)  # non-blocking

NaN-Safe Field Parsing

yfinance often returns float('nan') for missing fields (no trades, zero volume). nan is truthy in Python, so nan or 0 returns nan, and int(nan) raises ValueError. The _fi() / _ff() helpers handle this:

def _fi(val, default=0) -> int:
    try:
        v = float(val)
        return default if (math.isnan(v) or math.isinf(v)) else int(v)
    except (TypeError, ValueError):
        return default

Chain Enrichment

After fetching raw contract data from yfinance, enrich_chain() runs the full quant pipeline:

  1. Solve IV for each contract using market bid/ask midpoint
  2. Compute all five Greeks analytically
  3. Mark ATM strike (closest to current spot price)
  4. Compute chain statistics: ATM IV, HV-30d, PCR, OI exposure
  5. Persist to PostgreSQL (if available)
  6. Cache result in Redis with 55-second TTL

3.5 Strategy Builder

Long Call
+1 call @ ATM
Bullish. Unlimited upside, premium is max loss. Ideal when expecting a large up move.
Long Put
+1 put @ ATM
Bearish. Unlimited downside capture, premium is max loss. Hedge against long stock.
Bull Call Spread
+1 call ATM, −1 call OTM
Defined-risk bullish. Reduces cost vs naked call, caps upside at short strike.
Bear Put Spread
+1 put ATM, −1 put OTM
Defined-risk bearish. Cheaper than naked put, profit capped at short strike.
Straddle
+1 call ATM, +1 put ATM
Long volatility. Profits if stock moves significantly in either direction. Theta-heavy.
Strangle
+1 call OTM, +1 put OTM
Cheaper than straddle. Requires larger move to profit. Good before major events.
Iron Condor
−1 put OTM, +1 put far OTM
−1 call OTM, +1 call far OTM
Short volatility, range-bound. Max profit if stock stays between short strikes.
Butterfly
+1 call low, −2 call mid, +1 call high
Peaked payoff. Maximum profit if stock closes exactly at middle strike at expiry.

Payoff Calculation

def calculate_payoff(legs, price_range):
    prices  = np.linspace(price_range[0], price_range[1], 200)
    payoffs = np.zeros(len(prices))
    for leg in legs:
        intrinsic = np.maximum(0,
            (prices - leg.strike) if leg.option_type == 'C'
            else (leg.strike - prices))
        sign = 1 if leg.action == 'BUY' else -1
        payoffs += sign * leg.quantity * intrinsic - sign * leg.premium * 100
    return [PayoffPoint(price=p, pnl=v) for p, v in zip(prices, payoffs)]

3.6 Trade Analysis (Vol-Floor Model)

Inspired by Grid Reaper / ScaleOrder scaling strategies. Uses implied volatility to define a statistically-derived support zone for scaling into positions.

Floor Value = Entry Price × ( 1 − IV × √Period )

Example: SPY=$500, IV=20%, Monthly period:
Floor = 500 × (1 − 0.20 × √(1/12)) = 500 × 0.9423 = $471.15

This is a 1-standard-deviation downside move for the period. The grid levels are spaced equally between entry and floor, with capital allocated across levels to achieve a target profit if price recovers to entry.

SSR Disabled Globally

SvelteKit defaults to server-side rendering (SSR), which pre-renders page HTML on the server before sending it to the browser. Our pages make fetch('/api/...') calls with relative URLs. On the server, there is no host to resolve relative URLs against, causing a crash. Setting ssr = false in the root +layout.ts turns off SSR for the entire app:

// src/routes/+layout.ts
export const ssr = false;   // all pages render client-side only

4.2 Reactive Stores

Svelte stores are observable state containers. Any component can subscribe to them; when the value changes, all subscribers re-render automatically. The $ prefix is syntactic sugar for .subscribe().

// src/lib/stores/market.ts
export const selectedSymbol = writable<string>('SPY');
export const optionChain    = writable<OptionChain | null>(null);

// derived store — recalculates when optionChain or quote changes
export const atmStrike = derived(
  [optionChain, quote],
  ([$chain, $quote]) => {
    if (!$chain || !$quote) return null;
    return $chain.strikes
      .map(s => s.strike)
      .reduce((prev, curr) =>
        Math.abs(curr - $quote.price) < Math.abs(prev - $quote.price) ? curr : prev
      );
  }
);

In a component, using $selectedSymbol automatically subscribes and triggers re-render on change. No manual subscription management needed.

Refresh Timer Store

The refresh.ts store manages two intervals: one that counts down from 60 to 0 every second (for the navbar ring), and one that fires every 60 seconds to trigger a data reload. Both are guarded with if (!browser) return so they never run during SSR.

const BASE = '/api/v1';

async function request<T>(path: string, options?: RequestInit): Promise<T> {
  const res = await fetch(`${BASE}${path}`, {
    headers: { 'Content-Type': 'application/json' },
    ...options,
  });
  if (!res.ok) {
    const err = await res.json().catch(() => ({ detail: res.statusText }));
    throw new Error(err.detail ?? `HTTP ${res.status}`);
  }
  return res.json();

4.5 Charts (ECharts)

All charts follow the same pattern: onMount initializes the ECharts instance (canvas or WebGL), sets the option object, and a ResizeObserver handles responsive resizing. Reactive statements re-render when props change.

Svelte 5 onMount Pattern
In Svelte 5, onMount(async () => { return cleanup }) does NOT call the cleanup function because Svelte ignores Promise<cleanup>. All chart components use a synchronous onMount that launches async work via an IIFE and returns cleanup synchronously. 
// Correct pattern for all chart components
onMount(() => {
  let ro: ResizeObserver;
  (async () => {
    const echarts = await import('echarts');    // dynamic import
    chart = echarts.init(chartEl, null, { renderer: 'canvas' });
    ro = new ResizeObserver(() => chart.resize());
    ro.observe(chartEl);
    await loadData();
  })();
  return () => { if (ro) ro.disconnect(); if (chart) chart.dispose(); };
});

4.6 Theme System

The theme automatically switches based on Indian Standard Time (IST):

  • Light mode: 09:00 – 15:30 IST (Indian market hours)
  • Dark mode: All other times (pre-market, after-market, weekends)
  • User override: Stored in localStorage as ce_theme. If present, auto-switching is disabled.
// src/lib/stores/theme.ts
function getISTHour(): number {
  const now   = new Date();
  const utcMs = now.getTime() + now.getTimezoneOffset() * 60_000;
  const istMs = utcMs + 5.5 * 3_600_000;   // UTC+5:30
  return new Date(istMs).getHours() + new Date(istMs).getMinutes() / 60;
}

function inferThemeFromIST(): Theme {
  const h = getISTHour();
  return (h >= 9 && h < 15.5) ? 'light' : 'dark';
}

The theme store calls document.documentElement.classList.add('dark') or remove('dark') which triggers Tailwind's dark-mode class strategy. Every minute, the layout's setInterval calls theme.syncToIST() to re-check.

This chapter documents every non-trivial bug encountered during development, its root cause, and the fix. Understanding these bugs builds deep knowledge of Python, JavaScript, Svelte 5, and Vite internals.

Bug 1 — lib/ Gitignored

AspectDetail
SymptomAll frontend/src/lib/ files absent from git history
Root CausePython's default .gitignore includes lib/ (for virtualenvs). Git silently ignored the entire frontend source directory.
FixRemoved lib/ line from .gitignore

Bug 2 — ECharts .default Does Not Exist

AspectDetail
SymptomCharts blank; TypeScript: Property 'default' does not exist
Root CauseECharts uses named exports, not a default export. (await import('echarts')).default is undefined, so echarts.init() throws undefined is not a function.
Wrongconst echarts = (await import('echarts')).default;
Fixconst echarts = await import('echarts');

Bug 3 — onMount Async Cleanup Ignored

AspectDetail
SymptomResizeObserver / ECharts instance never cleaned up on component unmount; memory leak
Root CauseSvelte's onMount expects either () => void (sync cleanup) or Promise<never>. An async function returning a cleanup function produces Promise<() => void> — Svelte ignores it entirely.
FixNon-async onMount launches an async IIFE internally, returns cleanup synchronously.

Bug 4 — float('nan') or 0 Returns nan

AspectDetail
SymptomGET /api/v1/options/chain/AAPL → 500 Internal Server Error
Root Causeyfinance returns float('nan') for missing volume/OI. In Python, nan is truthy, so nan or 0 evaluates to nan, and int(nan) raises ValueError: cannot convert float NaN to integer.
FixAdded _fi()/_ff() helpers using math.isnan() before casting.

AspectDetail

Bug 7 — Blank Page (Root Cause: Wrong Svelte Build Served to Browser)

Most Complex Bug in This Project
This was the hardest bug to diagnose. The app silently produced a completely blank page with no visible errors. Required Chrome headless to capture the console error, then reading Svelte 5's internals to understand it.
AspectDetail
SymptomCompletely blank page in browser. No HTML rendered. No visible error.
Console ErrorUncaught TypeError: Cannot read properties of null (reading 'r') at svelte.js:144
Root CauseSvelte 5 ships two builds: index-client.js (browser) and index-server.js (server). The package.json exports use the "browser" condition to select the right one. SvelteKit's Vite plugin was not passing the browser condition to esbuild during dependency pre-bundling, so esbuild used the "default" export — which maps to index-server.js. The server build accesses ssr_context.r, but ssr_context is null in a browser environment. Every component crashed immediately. The bootstrap Promise chain swallowed the error silently.
FixExplicitly set resolve.conditions and optimizeDeps.esbuildOptions.conditions in vite.config.ts to ['browser', 'module', 'svelte', 'default'].
 
// vite.config.ts — the fix
export default defineConfig({
  plugins: [sveltekit()],
  resolve: {
    conditions: ['browser', 'module', 'svelte', 'default'],
  },
  optimizeDeps: {
    include: ['svelte', 'echarts'],
    esbuildOptions: {
      conditions: ['browser', 'module', 'svelte', 'default'],
    },
  },
});

Why ssr_context.r?

In Svelte 5's server build (index-server.js), the onDestroy function reads:

function onDestroy(fn) {
  ssr_context.r.on_destroy(fn);   // line 144 — ssr_context is null in browser!
}

ssr_context is a server-side global that tracks component lifecycle during HTML rendering. It is never initialized in a browser environment, so it is null, and accessing null.r throws the TypeError. The error is caught by the SvelteKit bootstrap Promise.all().then() but there is no .catch() handler, so the page silently stays blank.

8.3 Why NumPy is ~500× Faster

Vectorization Explained
A Python for-loop prices one contract per iteration. Each iteration has Python interpreter overhead (~50ns per bytecode). For 10,000 contracts, that's millions of bytecodes.

NumPy replaces the Python loop with a single call to a C extension that operates on entire arrays. The C code runs at CPU speed with SIMD (Single Instruction Multiple Data) instructions — one CPU instruction processes 4–8 floats simultaneously. Result: the same computation runs ~500× faster.

8.5 Why Svelte is Faster than React/Vue

React and Vue use a Virtual DOM (VDOM) — they maintain a JavaScript representation of the DOM, diff it on every update, and apply minimal patches. This diffing has non-zero overhead on every state change.

Svelte compiles components to vanilla JavaScript at build time. There is no VDOM, no diffing runtime, no framework overhead at runtime. A $selectedSymbol = 'NVDA' assignment compiles directly to a DOM update instruction. This is why Svelte re-renders are measured in microseconds.
